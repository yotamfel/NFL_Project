# NFL_project — Database Schema Reference

PostgreSQL 17. 15 objects: 9 base tables + 6 derived views, all in `public`.
Generated by querying `information_schema`/`pg_catalog` directly against the
live database — this reflects exactly what's there today (2000–2025, fully
filled). For the *why* behind any design choice mentioned here, see
`PROJECT_LOG.md`, which this document deliberately doesn't repeat.

## 1. The shape of it: a hub with seven spokes, plus one unlinked satellite

```
                         ┌────────────────────────┐
                         │        players         │  ← canonical identity
                         │  (player_id PK, name,  │    one row per PFR player
                         │   pos, season span)    │
                         └───────────▲────────────┘
                                      │ FK (player_id), nullable
            ┌─────────────────────────┼─────────────────────────────┐
            │                          │                             │
   passing_seasons ───► passing_career │                  combine_seasons
   offense_seasons ───► offense_career │                       draft
   defense_seasons ───► defense_career │
   kicking_seasons ───► kicking_career │
   punting_seasons ───► punting_career │
   returns_seasons ───► returns_career │
```

- **`players`** is the hub every other table joins against. It is *built
  from* the seven `*_seasons` tables (one collapsed identity per PFR
  `player_id`), not the other way around — see §6 for what that implies
  about query order and trust.
- The **seven box-score categories** each follow an identical two-table
  pattern: a `*_seasons` table (one row per player-season-team) and a
  `*_career` view (one row per player, derived by aggregation — see §3).
- **`draft`** and **`combine_seasons`** both carry an FK to `players`, but
  *not* in the direction you might expect — they describe events
  (a draft pick, a pre-draft workout) that may or may not correspond to a
  tracked career, so the FK is nullable both ways: many rows have no
  `player_id` (no tracked box-score career), and `players` has no inverse
  reference back to either.

## 2. `players` — the canonical identity table

| column | type | notes |
|---|---|---|
| `player_id` | text, **PK** | PFR's own stable id (e.g. `BradTo00`). ~80 older players carry an all-lowercase legacy format (`vinatada01`) — preserved as-is, confirmed valid. |
| `player_name` | text | the player's *most recently listed* name across all appearances (handles mid-career name changes, e.g. id `AlleJo03` = "Josh Allen" → "Josh Hines-Allen") |
| `pos` | text | the *most frequently listed* position — a single best-guess label, deliberately not a normalized taxonomy (collapsing e.g. "LDE"/"RDE" would lose real information) |
| `first_season`, `last_season`, `n_seasons` | bigint | span of *tracked* appearances across all seven categories — **not** necessarily a player's true rookie/final year (an offensive lineman may go years without a single tracked "skill" stat; see §6) |

11,627 rows, spanning seasons 2000–2025. Index: `ix_players_name`.

## 3. The seven box-score categories

Each category ships as a `*_seasons` table (raw, one row per player per team
per season — a player traded mid-season has multiple rows) and a `*_career`
view (one row per player, lifetime).

**Common `*_seasons` columns** (present in all seven, beyond category-specific stats):

| column | meaning |
|---|---|
| `id` | surrogate PK (bigint identity). Needed because `player_id` can legitimately be null (an unlinked player still has a real season row), and a natural-key constraint can't be a primary key when one of its columns is nullable. |
| `season` | year (int) |
| `rk` | PFR's own row-rank *within that season's table* — not meaningful as a cross-season/cross-category measure |
| `player_name`, `age`, `team`, `pos`, `g`, `gs` | self-explanatory (games played / started) |
| `awards` | free-text honors string straight from PFR (e.g. "AP MVP-1, PB, AP-1") |
| `player_id` | FK → `players.player_id`, nullable |

Constraints on every `*_seasons` table: `PRIMARY KEY (id)`,
`UNIQUE (player_id, season, team)` — except `combine_seasons`, which has no
team and uses `UNIQUE (player_id, season)` — `FOREIGN KEY (player_id)
REFERENCES players`, and an index on `season`.

**`*_career` views** collapse each player's seasons into lifetime totals —
*not* a blind `sum()`. Each column is classified by what its name signals:

- **counting stats** (yards, touchdowns, tackles, attempts, games, …) → **summed**
- **"longest play" columns** (`lng`, `rec_lng`, `rush_lng`, …) → **`max`** (a record, not a running total)
- **rates/percentages/averages** (`cmp_pct`, `y_per_a`, `qbr`, `koavg`, `fg_pct`, …) → **excluded entirely**. Summing a per-season rate across seasons of different lengths produces a meaningless number; recompute career rates from the summed counts at query time, e.g. `career_cmp_pct = 100.0 * sum(cmp) / sum(att)`.

Validated against Tom Brady's well-known career line (89,208 yards / 649 TD /
7,752-of-12,047) — exact match.

### 3a. `passing_seasons` / `passing_career`
2,749 / 802 rows · 96.3% linked

QB-specific columns worth knowing: `qbrec` (text `"W-L-T"` record string —
*recovered* from a Power-BI date-corruption, see `PROJECT_LOG.md`), `_1d`
(first downs — leading underscore because the source column name started
with a digit), `rate` (NFL passer rating), `qbr` (ESPN's Total QBR, only
populated in recent seasons — check for nulls before aggregating), `_4qc`
(4th-quarter comebacks), `gwd` (game-winning drives), `sk`/`sack_yds_lost`/
`sk_pct` (times sacked, not sacks made — contrast with `defense_seasons.sk`),
`y_per_a`/`ay_per_a`/`ny_per_a`/`any_per_a` (yards per attempt, in increasingly
adjusted forms — see PFR's glossary for the exact formulas, all reproduced
exactly in `supplement_seasons.py`).

### 3b. `offense_seasons` / `offense_career`
15,359 / 3,960 rows · 100% linked

Combined rushing + receiving (PFR's "Offense" page — covers RB/WR/TE/etc.).
Receiving block: `tgt`, `rec`, `rec_yds`, `rec_td`, `rec_first_downs`,
`rec_lng`, `ctch_pct`, `y_per_tgt`, `y_per_r`. Rushing block: `att`,
`rush_yds`, `rush_td`, `rush_first_downs`, `rush_lng`, `y_per_a`. Combined:
`touch` (`att + rec`), `yscm` (yards from scrimmage = `rush_yds + rec_yds`),
`rrtd` (rushing + receiving TDs), `fmb` (fumbles).

### 3c. `defense_seasons` / `defense_career`
38,826 / 9,396 rows · 96.6% linked — by far the largest table (every
defensive contributor in every game logs a row)

Two tackle-credit families that **don't always agree** across sources (PFR's
own page mixes in fringe players who never otherwise appear defensively, and
tackle-crediting is famously inconsistent league-wide — documented, not a bug
in this pipeline): `comb`/`solo`/`ast`/`tfl`/`qb_hits`. Turnover-producing
plays: `int`, `int_ret_yds`, `int_td`, `pd` (passes defended), `ff` (forced
fumbles), `fr`/`fr_td`/`fum_ret_yds` (fumble recoveries — note `fum_ret_yds`
credits whoever *recovered* the ball, regardless of whose fumble it was —
see `PROJECT_LOG.md` for why that convention was chosen). `sk` here means
sacks **made** (inverse of `passing_seasons.sk`). `sfty` = safeties.

### 3d. `kicking_seasons` / `kicking_career`
1,678 / 339 rows · 100% linked

Field goals are bucketed by distance: `fga_0_19`/`fgm_0_19` … through
`fga_50_plus`/`fgm_50_plus`, plus `fga_total`/`fgm_total`/`fg_pct`. A useful
identity to sanity-check any slice of this data against:
`fga_total = Σ fga_*` (PFR counts a *blocked* attempt at the distance it was
kicked from — see `PROJECT_LOG.md` for how `supplement_seasons.py` parses
`fg_blocked_list` to keep this holding for 2025+). Extra points: `xpa`/`xpm`/
`xp_pct`. Kickoffs: `ko`/`koyds`/`koavg`/`tb`/`tb_pct`.

### 3e. `punting_seasons` / `punting_career`
1,123 / 237 rows · 100% linked

`pnt` (punts), `yds`/`y_per_p` (gross), `retyds` (return yards allowed),
`netyds`/`ny_per_p` (net — note the identity `netyds = yds − retyds`), `tb`/
`tb_pct` (touchbacks), `pnt20`/`in20_pct` (downed inside the 20), `blck`
(blocked).

### 3f. `returns_seasons` / `returns_career`
5,921 / 2,649 rows · 100% linked

Punt-return and kick-return blocks (`punt_ret`/`kick_ret` × `_yds`/`_td`/
`_lng`/`y_per_*_ret`), plus `apyd` (all-purpose yards — a wider figure than
"yards from scrimmage + return yards": it also folds in the player's *own*
interception-return and fumble-return yardage when they also play defense;
see `PROJECT_LOG.md` for the Marcus Jones proof).

## 4. `draft` — annual draft picks, linked back to careers where possible

6,644 rows · 2000–2025 · 92.0% linked (`UNIQUE (draft_year, round, pick)`,
`FOREIGN KEY (player_id) REFERENCES players`, index on `player_id`)

| column | meaning |
|---|---|
| `draft_year`, `round`, `pick`, `team` | the pick itself |
| `player_name`, `pos`, `age`, `college` | as listed at draft time (may differ slightly from `players` — that's expected, see §2) |
| `last_season`, `all_pro_yrs`, `pro_bowls`, `years_started`, `career_av`, `draft_team_av`, `g` | PFR's own career-summary columns *as of when the draft page was scraped* — a snapshot, not a live recomputation from `*_seasons` |
| `pass_*`, `rush_*`, `rec_*`, `solo_tkl`, `def_int`, `sk` | a small fixed set of headline career stats PFR shows directly on the draft page (not the full box-score detail — for that, join to the linked player's `*_career` view) |
| `player_id` | FK → `players`, **null for ~8% of picks** — see §6 before treating null as "never played" |

**Important nuance for analysis**: the `pass_*`/`rush_*`/etc. columns here are
a *PFR-curated snapshot at draft-page-scrape time*, not a live join. For any
serious "how did this draft class perform" analysis, prefer joining
`draft.player_id → players.player_id → *_career` for live, consistently
aggregated figures, and treat this table's own stat columns as a quick
cross-check / fallback for unlinked players only.

## 5. `combine_seasons` — pre-draft workout measurements

8,649 rows · 2000–2025 · 81.6% linked (`UNIQUE (player_id, season)`,
`FOREIGN KEY (player_id) REFERENCES players`, index on `season`)

| column | meaning |
|---|---|
| `season` | the combine class year (== draft year for drafted attendees) |
| `player_name`, `pos`, `school` | prospect identity (no `age` column — that lives on `draft`, for picks that have one) |
| `ht` | height as **`"feet_inches"`**, e.g. `"6_2"` — ⚠️ the underscore is PFR's own raw-export separator convention, confirmed *not* a corruption (zero hyphenated values exist anywhere in the column); cast carefully if you need inches (`split_part(ht,'_',1)::int*12 + split_part(ht,'_',2)::int`) |
| `wt` | weight in pounds |
| `_40yd`, `_3cone`, `shuttle` | drill times in seconds (leading underscores: source names started with digits) |
| `vertical`, `broad_jump` | jump distances in inches |
| `bench` | bench-press reps at 225 lbs |
| `college` | ⚠️ **mostly noise** — either null or the literal placeholder text `"College Stats"` (PFR's link-label artifact); use `school` instead |
| `drafted_tm_per_rnd_per_yr` | free-text draft-slot string, format `"{team} / {round}{suffix} / {pick}{suffix} pick / {year}"`, parsed by `link_draft.py`'s regex to cross-reference against `draft` — null for undrafted attendees |
| `player_id` | FK → `players`, null for prospects with no tracked box-score career (~18%) |

## 6. Things that will trip up an analysis if you don't know them going in

These aren't bugs — each is investigated and documented in `PROJECT_LOG.md`.
Listed here because they're the kind of thing that looks like a data-quality
problem the moment you start querying:

1. **A null `player_id` does not mean "never made a roster."** It means "no
   *box-score* stat was ever tracked for this PFR id" — true for ~8% of
   draftees and ~18% of combine attendees, overwhelmingly offensive linemen
   and other no-personal-stat positions with real, lengthy careers (e.g. the
   2025 #4-overall pick, LSU tackle Will Campbell, appears in *zero* tables
   despite being a starting NFL tackle as a rookie). Don't filter these out
   as "busts" — check `g` in `draft` for actual games played.
2. **`players.first_season` can sit years after a player's true rookie
   season**, for the same reason — their tracked history only starts the year
   they happen to record a fluke "skill" stat (a recovered fumble, a
   trick-play catch). Don't use it as a proxy for "draft-to-debut gap."
3. **Season-level rate columns must never be summed or averaged across
   seasons** — that's exactly why the `*_career` views exclude them. If you
   need a career rate, recompute it from summed counts.
4. **`combine_seasons.ht` uses underscores, not hyphens** — don't
   "helpfully" convert it back; that would actually be introducing the
   corruption pattern this project spent real effort *reversing* elsewhere
   (see `corruption_fixes.py` / `PROJECT_LOG.md` for the genuine
   hyphen→underscore corruption in player names, which is a different thing
   entirely from this column's native format).
5. **Tackle counts (`comb`/`solo`/`ast`) and `tgt`** carry known,
   investigated-and-accepted cross-source noise (different conventions for
   penalty-nullified plays, fringe-player inclusion, etc.) — don't chase
   "why doesn't this exactly match [other source]" for these specific columns.
6. **`draft`'s own career-stat columns (`pass_yds`, `career_av`, …) are a
   scrape-time snapshot**, not a live aggregation — they will silently drift
   out of sync with `*_career` for any player who's kept playing since. Prefer
   the linked join for anything beyond a quick lookup.
7. **Query/build order matters if you ever touch `players`.** It's *built
   from* the seven `*_seasons` tables (`build_players.py`'s full rebuild, or
   `supplement_players.py`'s incremental upsert) — not the other way round.
   A brand-new player must be seeded into `players` *before* their first
   season row can be written (the FK requires it); `supplement_seasons.py`
   handles this automatically via `_seed_new_players`, but any ad hoc insert
   into a `*_seasons` table must respect the same order.

Every table now covers the identical 2000–2025 range — the last gaps (the
2025 draft class and its combine) were closed in the most recent commit.

tables this is "share of player-seasons traceable to a canonical identity",
for `draft`/`combine_seasons` it's "share of picks/prospects with a tracked
box-score career" (see §6, point 1, before reading too much into the gap).
